Boss Wave Spawner

An editor-configurable wave spawner for boss encounters — define enemy types and counts per wave without touching code, built on an in-house ECS engine.

Overview

This system lets designers configure boss encounter waves entirely through the editor. Each wave can have its own mix of enemy types and quantities — no code changes needed to tweak a fight. The spawner picks the best spawn point at runtime and works its way through the wave queue automatically.

It's built on Goose, an in-house ECS engine used at TGA, and uses a BossWaveSpawnerComponent that stores all the wave data and runtime state together on the entity.

Architecture

The system is split across three pieces:

BossWaveDataReferenceProperty

Pure data — small spider, big spider, and drone counts per wave. Editable in the inspector.

BossWaveSpawnerComponent

Owns the wave list, spawn point refs, prefab refs, and live runtime state like current wave index and the active spawning queue.

Boss Spawner System

Each tick, iterates the spawner view, picks the furthest spawn point, and instantiates the next enemy in the current wave.

Enemy Types

Three enemy types can be mixed freely across any number of waves:

Small Spider

Lightweight, spawns first in each wave to pressure the player early.

Big Spider

Tankier variant, spawns after small spiders in the wave order.

Drone

Spawns last per wave. Each type is pulled from a configured prefab reference.

Spawn Point Selection

Rather than picking a spawn point at random, the system calls GetFurthestSpawnPointIndex each time it needs to place an enemy. This keeps spawns away from where the player currently is, which avoids enemies appearing directly on top of them mid-fight.

Wave Queue

Active waves are tracked in a spawningWaves vector. Each tick the system removes any wave from the front of the queue whose enemy counts have all hit zero, then spawns the next enemy from the current wave. When the queue empties, isSpawning is cleared. 

// Finished waves are pruned before spawning
comp.spawningWaves.erase(
    std::remove_if(comp.spawningWaves.begin(), comp.spawningWaves.end(),
        [&comp](const int& aWaveIndex) {
            const auto& wave = comp.bossWaves.at(aWaveIndex);
            const int remaining = wave.bigSpiderCount
                               + wave.smallSpiderCount
                               + wave.droneCount;
            return remaining <= 0;
        }),
    comp.spawningWaves.end()
);

Editor Integration

The component is fully reflected via OnReflectComponent, so all wave data, prefab references, and spawn point links show up directly in the engine inspector. A designer can add waves, set counts, and swap prefabs without opening a single source file.

What I learned

Getting the wave queue cleanup right took a bit of thought — the erase-remove idiom is clean but you have to make sure the counts are being mutated on the stored wave, not a copy. I also found that spawning enemies one per tick (rather than all at once) gave a much better feel in-game and avoided hitches from instantiating many prefabs in a single frame.

If I were extending it further I'd add per-wave spawn delays and a way to trigger waves based on boss health thresholds rather than a manual start flag.
