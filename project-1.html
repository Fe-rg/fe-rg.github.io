Ability Creation Tool

A two-part system for designing and stacking projectile modifiers in an ImGui editor, with JSON save/load and full runtime behavior — no code changes needed to build a new ability.

Overview

For my specialization in 2nd year at TGA I wanted to build something inspired by Path of Exile's support gem system where you can stack modifiers onto an ability to completely change how it behaves. Things like making projectiles chain between enemies, split apart on hit, or fork into multiple directions on collision.

The goal was that a designer could open an ImGui window, pick and tune modifiers, reorder them, save the result to JSON, and load it back later, without touching any code. The runtime then reads that setup and plays it back in-game.

Part 1 — The Editor

The ImGui editor lets you toggle modifiers on and off, configure their settings with drag sliders and int inputs, and reorder them with up/down buttons since the order they fire in actually matters. When you're happy with the setup you hit save, which writes each modifier's data out to a JSON file. There's also a directory scan that finds all existing ability files so you can pick one from a dropdown and load it back. 

// Each modifier handles its own save/load
inline void Fork::SaveJSon(const char* aFilePath, const char* /*aName*/) {
    JSonManager::Save(aFilePath, "Fork", myForkCount,    "ForkCount");
    JSonManager::Save(aFilePath, "Fork", myForkAngle,    "ForkAngle");
    JSonManager::Save(aFilePath, "Fork", myForkProjectileCount, "ForkProjCount");
}

Part 2 — Modifiers

All modifiers share a virtual Modifier base with an Apply(Projectile&) method, a Clone() for copying onto new projectiles, and SaveJSon/LoadJSon. There are two categories:

ProjectileSpeed

Base modifier. Adds a flat speed bonus when the ability fires — applied immediately, never queued.

MoreProjectiles

Base modifier. Increases the projectile count fired in the spread. Applied before any are spawned.

Chain

On collision, finds the nearest enemy that wasn't just hit and redirects the projectile toward it.

Fork

On collision, disables the projectile and spawns N child projectiles spread across a configurable angle.

Split

Like Fork but keeps the parent alive, spawning children alongside it. Supports multiple split charges.

Orbit

Changes the projectile's travel type to orbit the player instead of traveling in a straight line.

Here's Fork::Apply as an example. It uses a quaternion to rotate the parent's direction by even steps across a configurable angle, spawns each forked child projectile with its own copy of the remaining modifier queue, then disables the parent: 

bool Fork::Apply(Projectile& aParentProj) {
    if (myForkCount > 0) {
        myForkCount--;

        for (int i = 0; i < myForkProjectileCount; i++) {
            // Spread evenly across the fork angle, centered on parent direction
            const float angle = -myForkAngle * 0.5f
                + (myForkAngle / static_cast<float>(myForkProjectileCount - 1)) * static_cast<float>(i);

            Tga::Quaternionf rot({ 0.f, 1.f, 0.f }, angle);
            Tga::Vector3f dir = Tga::Quaternionf::RotateVectorByQuaternion(
                rot, aParentProj.GetDirection()).GetNormalized();

            Projectile forkedProj;
            forkedProj.Init(aParentProj.GetProjectileSpeed(), aParentProj.GetPosition(),
                dir, aParentProj.GetModifiers(), aParentProj.GetActiveModifiers(),
                aParentProj.GetLastHitEnemy());

            aParentProj.AddChildProj(std::move(forkedProj));
        }

        aParentProj.DisableProjectile();  // parent dies, children carry on
        return true;
    }
    return false;
}

Chain works differently, instead of spawning new projectiles it finds the closest enemy within a max distance and redirects the existing projectile towards it. Chain only returns true if there are still charges left after redirecting, so the projectile knows when it's done chaining: 

bool Chain::FindTarget(Projectile& aParentProj) {
    if (myChainCount <= 0) return false;

    float closestDist = std::numeric_limits<float>::max();
    Tga::Vector3f closestPos;
    bool foundTarget = false;

    for (auto& it : Getter::GetGameWorld()->GetEnemies()) {
        const float dist = Tga::Vector3f::Distance(
            it->transform.GetPosition(), aParentProj.GetPosition());

        // Skip the enemy we just hit, and anything out of chain range
        if (it == aParentProj.GetLastHitEnemy() || dist > myMaxChainDist) continue;

        if (dist < closestDist) {
            closestDist = dist;
            closestPos  = it->transform.GetPosition();
            foundTarget = true;
        }
    }

    if (foundTarget)
        aParentProj.SetDirection(newDirection(aParentProj.GetPosition(), closestPos));

    --myChainCount;
    // Return false on the last charge so the projectile knows it's done chaining
    return myChainCount > 0 && foundTarget;
}

Part 3 — Runtime

Each projectile carries a copy of the modifier map and an ordered list of which active modifiers are still in play. Base modifiers (speed, count) get consumed immediately on Init — they're not queued, they just modify the starting state and get removed. 

// Base modifiers fire immediately on Init and are stripped from the queue
for (size_t i = 0; i < myActiveModifiers.size();) {
    auto* mod = myModifiers[myActiveModifiers[i]].get();
    if (mod && mod->IsBaseModifier()) {
        mod->Apply(*this);
        myActiveModifiers.erase(myActiveModifiers.begin() + i);
        continue;
    }
    ++i;
}

The remaining active modifiers sit in a queue and fire on collision, in the order you set in the editor. The same modifier keeps firing on each collision until it runs out of charges, then the next one in line takes over. When the queue is empty and the projectile collides again, it deactivates. 

// On collision -> apply the front modifier, pop it if it's out of charges
const ModifierTypes type = myActiveModifiers.front();
Modifier* mod            = myModifiers[type].get();

if (mod) {
    // Apply returns false when the modifier is exhausted
    if (!mod->Apply(*this)) {
        myActiveModifiers.erase(myActiveModifiers.begin());
    }
}

Child projectiles (from Fork or Split) are stored on the parent and updated recursively each tick, so a single fired projectile can branch into a whole tree of children that all clean themselves up when done. 

// Returns true while any child is still alive, keeping the parent alive too
bool Projectile::UpdateChildren(const float aDeltaTime) {
    for (size_t i = 0; i < myProjChildren.size();) {
        if (!myProjChildren[i].Update(aDeltaTime))
            myProjChildren.erase(myProjChildren.begin() + static_cast<int>(i));
        else
            ++i;
    }
    return !myProjChildren.empty();
}

// Parent stays alive as long as it or any child is still active
const bool isActive = UpdateChildren(aDeltaTime) || myActive;
return isActive;

What I learned

The trickiest part was getting the modifier queue to feel right, especially making sure base modifiers were consumed at init time rather than on the first collision, and that the clone on each new projectile was deep enough that charges on one projectile didn't affect another.

If I were to extend it I'd add support for condition-based triggers beyond just collision (like a timer or a distance threshold).

One thing I regret doing was using the schools own in-house engine instead of the one me and my group have made for our projects, as the schools engine is rather lacking as we're supposed to develop a lot of it ourselves for said projects. This caused me to spend time solving certain issues like collision that could've been spent doing other things which led to an overall worse product in the end.
