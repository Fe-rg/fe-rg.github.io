Player Ability System

A data-driven ability framework built on an in-house engine. Supporting multiple ability types with per-type configuration, cooldowns, and JSON serialization.

Overview

This system was built during a game project at TGA, using an in-house engine with Diablo 3 as reference. The goal was a clean, extensible way to give the player multiple distinct abilities each with its own data, cooldown, and runtime behavior without hardcoding ability logic into the player component itself.

Each ability type carries its own configuration struct and can be set up, serialized, and tweaked through an ImGui editor window at design time, then loaded back at runtime via JSON.

Architecture

The system is split across three layers:

AbilityBase (config)

Pure data structs per ability type. Handles ImGui editing, JSON save/load.

PlayerAbility (runtime)

Wraps config with live state — cooldown timer, unlock flag etc.

PlayerComponent (ECS)

Owns a map of abilities keyed by state enum.

Ability types are identified by a PlayerStates enum, which doubles as the state machine state the player enters when using that ability. This means the ability system and the state machine stay in sync by design, no extra mapping needed.

Ability Types

Five concrete ability types each inherit from AbilityBase and define their own fields:

Dash

Max distance + speed. Moves the player rapidly in a direction.

Shield

Duration only. Activates a timed defensive state.

Spell

Radius, damage, hitbox frame. A general-purpose AoE cast.

Generator

Builds a resource on hit. Same shape as Spell but distinct behavior.

Spender

Consumes built-up resource for an empowered attack.

JSON Serialization

Every ability type implements LoadJson and SaveJson via the virtual AbilityBase interface, so the editor can save any ability to disk and the game can reload it at startup without knowing the concrete type at the call site. 

// Each ability type handles its own serialization
void DashAbility::LoadJson(const nlohmann::json& aJson) {
    length = aJson.at("length").get<float>();
    speed  = aJson.at("speed").get<float>();
}

// Runtime setup picks the right concrete type from the enum
case PlayerStates::Dash:
    return new Dash(static_cast<const DashAbility&>(aAbility));

ImGui Integration

Each ability type also implements ImGuiSetup, which renders the relevant drag fields directly in the editor. A designer can open the window, pick an ability type, tune the values visually, and save — no recompile needed. The uint8_t& aInfo flag is used to detect when any value was changed so the editor knows to mark the file as dirty.

What I learned

This was my first time designing a system that needed to be both editor-friendly and runtime-efficient. Using a virtual base with per-type structs kept each ability's data isolated and easy to extend, adding a new ability type is just a new struct and a case in the factory switch.

If I were to revisit it I'd look at replacing the raw new/delete pattern with std::unique_ptr to make ownership clearer and avoid having to remember the manual cleanup in the destructor.
